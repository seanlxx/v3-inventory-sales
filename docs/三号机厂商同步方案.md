## 0. 背景与范围

- 用户在第三方售货机平台 `mv.smshj.com/hbshengma`（下称"盛码"）拥有一台售货机，本系统中命名为 **三号机**。
- 盛码无官方开放 API，但移动端 Web 后台有公开页面可供登录后访问，本方案通过**模拟登录 + 抓取页面/接口**的方式读取数据。
- **本期范围（只读同步）**：销售、库存、售价、成本从盛码同步到当前后台。
- **不在范围**：
  - 不回写盛码（不改盛码库存 / 售价 / 成本 / 机器配置）。
  - 只做"手动按钮 + 只读同步"。
  - 不接入其他三方平台（仅盛码一家）。

## 1. 盛码侧情报（已侦察）

### 1.1 登录流程

**入口**：`POST /mobile/mobilelogin.html`

**加密流程**（从 `/mobile/index.html` 页面 JS 逆向）：

1. 前端随机生成 16 字符 AES key（字符集 `[0-9a-zA-Z]`）。
2. AES-ECB-PKCS7 加密明文密码 → base64 → 表单字段 `password`。
3. RSA-PKCS1 用硬编码公钥加密 AES key → base64 → 表单字段 `encryptAesKey`。
4. POST 表单字段：`username` / `password` / `encryptAesKey` / `rememberMe=true`。
5. 服务端用私钥解出 AES key，再解出明文密码做校验，成功后下发 Cookie 维持会话。

```jsonc
    "status": "success",                          // running | success | failed
    "dryRun": false,
    "dateRange": { "start": "2026-05-15", "end": "2026-05-15" },
    "summary": {
      "salesImported": 12, "salesDuplicate": 3, "salesSkipped": 1,
      "productsCreated": 0, "pricesUpdated": 5, "costsUpdated": 5,
      "inventoryAdjusted": 8, "warnings": 2
    }
  }

```

#### `POST /api/integrations/shengma/sync`

**请求体**：

```jsonc
{
  "startDate": "2026-05-15",     // 必填，YYYY-MM-DD
  "endDate":   "2026-05-15",     // 必填，YYYY-MM-DD，区间 ≤ 31 天
  "dryRun":    false,            // 可选，默认 false。true 时只解析不写入
  "scope": ["inventory","sales"] // 可选，默认 ["inventory","sales"]
}
```

**响应**：同 `status.lastRun` 结构 + `warnings: string[]` 明细。

**约束**：

- 同一时刻只允许 1 个同步进程运行（基于 `external_sync_runs.status='running'` 锁，超时 5 分钟自动释放）。
- `endDate >= startDate`，区间 ≤ 31 天。
- `dryRun=true` 时，所有数据库写操作走事务并最后 rollback（或全程不开事务、只跑解析+计算）。

### 3.4 Cookie 缓存策略

- 登录成功后，把 Cookie + 过期时间存到 `app_settings` 表（key=`shengma.session`，value 为 JSON）。
- 每次同步前先尝试用缓存 Cookie 调一次轻量接口（如 `goods.html`）；若返回是登录页 → 重新登录。
- 目的：避免每次同步都登录一遍，触发风控。

## 4. 业务规则

### 4.1 商品归一化与匹配

**归一化函数 `normalizeProductName(name)`**：

1. 转小写
2. 去除空格、半/全角括号、中英文标点
3. 统一单位：`ML/毫升` → `ml`、`G/克` → `g`、`L/升` → `l`
4. 保留中文、数字、英文字母

**示例**：

| 原始 | 归一化结果 |
| --- | --- |
| `可口可乐 330ml` | `可口可乐330ml` |
| `可口可乐(330ML)` | `可口可乐330ml` |
| `Coca-Cola 330ML` | `cocacola330ml` |

**匹配逻辑**：

1. 先在 `products` 表中按 `machine_id=三号机` + `normalized_name` 精确匹配。
2. 匹配不到 → 自动创建：`name=原始名`、`normalized_name=归一化名`、`external_id=归一化名`。
3. 在 `external_product_mappings` 记录映射，便于后续人工查阅。
4. 前端"商品管理"页提供手动合并入口（本期不实现，留口子）。

### 4.2 售价与成本同步

- `products.sell_price_cents` ← 盛码当前售价。
- `inventory_balances.avg_cost_cents` ← 盛码当前进价。
- `inventory_balances.inventory_value_cents` ← `当前库存 × 同步后成本`。

**多货道聚合**：

| 维度 | 规则 |
| --- | --- |
| 库存 | 求和（所有货道相加） |
| 售价 | 取出现次数最多的价格；并列时取**最低价**（对用户友好），记 warning |
| 成本 | 按库存加权平均；库存全为 0 时取出现次数最多的成本，并列时取**最高值**（保守），记 warning |
| 不一致 | 任何不一致都写一条 warning 到 `external_sync_runs.warnings_json` |

### 4.3 忽略规则

以下数据**不导入**，只在 warning 中提示：

- 商品名包含 `请勿下单`、`测试`、`勿点` 的货道
- 隐藏货道（盛码页面标记为隐藏）
- 售价为 `999` 或 `999.00`（测试价占位符）

### 4.4 销售单导入规则

- 仅导入 `已支付 + 已出货 + 未退款` 的订单，其他状态跳过 + warning。
- 本地销售单 `external_id` = 盛码订单号；`source = 'shengma'`。
- 通过唯一索引 `(source, external_id)` 去重，重复同步不会插入重复单据。
- 销售收入：用盛码成交金额（含支付方式后的实收金额）。
- 销售成本（`sales_items.cogs_cents`）：
  1. 优先使用销售详情里的进价字段（如盛码返回）。
  2. 缺失时使用同步后的 `inventory_balances.avg_cost_cents`。
- 外部销售导入走专用服务 `importer.js` 中的 `importExternalSale()`，允许显式写入 `cogs_cents`，**不暴露给普通手工录入接口**。

### 4.5 库存校准（关键：用调整流水，不直接覆盖）

为遵循 v3 的"流水账本是唯一可信来源"设计，库存校准采用**调整模式**，不直接覆盖余额：

1. 算出本地当前 `inventory_balances.qty`（按 movement 流水重建）。
2. 拿到盛码当前货道库存合计 `vendorQty`。
3. 计算差值 `delta = vendorQty - localQty`。
4. 若 `delta != 0`，在 `stock_movements` 写一条：
   - `movement_type = 'adjustment'`
   - `qty_delta = delta`
   - `reason = '盛码同步校准 YYYY-MM-DD HH:mm'`
   - `external_id = 'shengma:adjust:<sync_run_id>:<product_id>'`（防止重复点击产生重复调整）
5. `inventory_balances` 由触发器/重算函数自动更新。

### 4.6 同步执行顺序

```
1. 登录 / 验证 Cookie
2. 抓 goods.html  → 解析货道、商品名、售价、库存
3. 抓 setJinjia.html → 解析进价
4. 抓 salesAll.html → 分页拉销售（按日期范围）
5. dryRun=false 时进入事务：
   ① 同步商品（创建 / 更新 normalized_name / external_id）
   ② 更新售价 sell_price_cents
   ③ 更新成本 avg_cost_cents
   ④ 写入快照 external_inventory_snapshots
   ⑤ 导入销售单（幂等）
   ⑥ 库存校准（写 adjustment 流水）
6. 写 external_sync_runs（status / summary / warnings）
```

> ⚠️ 必须先同步商品/售价/成本，再导入销售，最后校准库存。
> 若用户在前端只勾"销售" → 弹提示："建议先同步库存/成本，否则销售成本可能不准确"。

## 6. 风险清单

| # | 风险 | 概率 | 应对 |
| --- | --- | --- | --- |
| 1 | 盛码加验证码 / 滑块 | 中 | 同步失败 → 提示用户去盛码后台登录一次解锁，下次再试 |
| 2 | `node:crypto` 在 Pages Functions 不可用 | 低-中 | 兜底用 `crypto-js` + `jsencrypt`（项目已有依赖） |
| 3 | 盛码改接口/页面结构 | 高 | 解析失败 → 同步失败并提示具体字段，**不静默猜测导入** |
| 4 | 高频登录触发风控 | 中 | Cookie 缓存复用 + 同步间隔由用户控制 + 同一时刻只允许 1 个运行 |
| 5 | 商品名归一化误匹配（不同商品同名） | 低-中 | `external_product_mappings` 表保留原始名，前端可人工查 / 改 |
| 6 | 销售单重复导入 | 高 | `(source, external_id)` 唯一索引硬约束 |
| 7 | 库存校准把手工调整覆盖 | 中 | 用 `adjustment` 流水模式，不直接覆盖余额；历史调整完整保留 |
| 8 | 账号密码泄露 | 高 | 只走 Cloudflare Secrets，不入源码 / 文档 / Git |
| 9 | 合规：爬取第三方系统 | 低 | 用户自己的账号、自己的机器数据；若盛码 ToS 禁止则改为半自动导入路线 |

## 9. 假设与遗留

### 9.1 假设

- 本期只读，不写回盛码。
- 盛码接口在可预见的未来（3–6 个月）不会大改；如改动，同步失败时明确提示用户。
- 盛码不会因每次同步 3–4 个请求封禁账号；触发风控时改为每天手动一次。
- `node:crypto` 在 Pages Functions 可用；否则兜底用 `crypto-js` + `jsencrypt`。
- 用户自己的账号 + 自己的机器数据，合规风险低。

### 9.2 遗留（本期不做）

- 商品名人工合并 UI（本期靠 `external_product_mappings` 表手动改）。
- 多店铺 / 多机器扩展（本期只接三号机一台）。
- 回写盛码（改库存 / 售价 / 上下架）。

### 9.3 待 POC 确认的开放问题

1. `salesAll.html` 返回是 HTML 还是 JSON？分页总数字段叫什么？
2. `goods.html` 的 `v=9` 参数含义？是否必须？
3. `setJinjia.html` 是 GET 读取还是 POST 设置？若仅为设置页面，进价的真实读取接口在哪？
4. 盛码销售详情里是否包含进价字段（用于销售成本回填）？
5. 退款 / 异常订单在销售列表里如何标识？
