# 1 号机 / 2 号机 销售-商品-库存 重构计划

> 适用范围：v3 售货机管理系统的 1 号机 / 2 号机（zn 平台 Excel 导入链路）。3 号机（盛码同步链路）只在金额口径与防漂监控上对齐，不动同步流程。
>
> 文档版本：2026-05-27 初版。任何阶段修改要回到本文件更新对应小节，不要散在多处。

## 0. 总目标与一句话价值

| 维度 | 现状 | 目标 |
| --- | --- | --- |
| 库存维度 | 1 / 2 号机被 `stockMachineIdFor` 折叠为虚拟 `'1/2号机'`，无法分机维护台账 | 台账按真实设备独立维护，可单机盘点、单机调拨；库存列表只显示商品总库存 |
| 进货-库存恒等式 | `Σ进货 ≠ 当前库存 + 累计出库`（漂移已客观存在） | 重建后等式成立，且 Phase 2.5 的 5 项对账每次部署前重跑都为 0 |
| 金额口径 | 销售额 / 实收 / 手续费 / 算法费 / 退款 / 成本 / 毛利的关系散落多处实现 | 全站统一公式，落到一份 `_shared/money.js` |
| Excel 导入 | 只处理 `订单明细`，结算字段（手续费、算法费、实收）依赖订单明细行级 | 双 Excel 协同：`订单明细` 是销售来源，`交易账单` 是结算来源，按订单号关联回写 |
| 余额更新逻辑 | 3 处副本（`inventory-service.js` / `zn/importer.js` / `shengma/importer.js`） | 合并为 `_shared/inventory-balance.js` 唯一实现 |

**一句话价值**：「台账按机分、页面看总库存、金额对得上、流水可重建、漂移看得见」。

## 1. 漂移根因清单（先盘清楚再动手）

按 code review 已知，至少有这些漂移源。Phase 2 必须把它们逐条消灭，否则重置后还会再漂。

| 编号 | 根因 | 位置 | 影响面 |
| --- | --- | --- | --- |
| R1 | 3 处余额更新副本算法不一致 | `_shared/inventory-service.js` `applyBalanceDelta` / `_shared/zn/importer.js` 余额段 / `_shared/shengma/importer.js` `applyAdjustmentToBalance` | 任意一处算错就漂；负库存恢复分支三家不一致 |
| R2 | `stockMachineIdFor` 折叠 1/2 → '1/2号机' | `_shared/stock-scope.js` | 历史上若有写入用了 `'1号机'` 而不是 `'1/2号机'`（或反过来），同一商品库存被劈成两行 |
| R3 | `inventory_balances` 是缓存而非真相 | schema 设计 | 写入未走原子批量时，movements 落库但 balance 没更新（或反之） |
| R4 | 负库存恢复分支按当前进价折算 | `applyBalanceDelta` 内 `previousQty < 0 && qtyDelta > 0` | `inventory_value_cents` 与历史购入累计成本对不上 |
| R5 | 作废未完整反冲 | `voidDocument` 漏写一条对应反向 movement | 漏一条漂一条 |
| R6 | adjustment movement 无凭据 | 任意手工/导入产生的 adjustment | `Σ进货 - Σ销售 ≠ 当前库存`，差额沉淀在 adjustment 里 |
| R7 | 同名重复商品 | `products` 唯一约束薄弱（同 machine_id 下 normalized_name 不唯一） | 自动建一份、手工又建一份，库存挂两边 |
| R8 | 结算字段不全 | `订单明细` Excel 不带平台清算实收，单纯靠预估到账 | `received_amount_cents` 偏差，毛利不准 |

> 引用编号 R1～R8 在后续阶段会反复出现，作为「这一步在解决哪条根因」的索引。

### 4.1 Phase 0.5 — 漂移诊断 + 现状文档（强制，重置前）

**目标**：把当前漂移定量化，留下「重建前」对照基线。重置后没人再回头看得到原始漂移。

**新增脚本**（放 `scripts/diagnose/`）：

| 脚本 | 校验内容 | 失败说明 |
| --- | --- | --- |

### 4.3 Phase 1 — 金额语义统一 + 结算导入（2d）

**目标**：把销售额 / 实收 / 手续费 / 算法服务费 / 优惠 / 退款 / 成本 / 毛利的口径锁死，并支持 `交易账单` Excel 导入回写实收。

**统一公式**（写入 `functions/api/_shared/money.js`，所有调用方禁止再自己拼）：

```
原始销售额  gross_amount_cents     = 销售额（订单金额，含优惠后）
退款金额    refund_amount_cents    = 平台退给买家
平台手续费  platform_fee_cents     = 平台抽佣（计入费用）
算法服务费  service_fee_cents      = 算法服务费（计入费用）
优惠金额    discount_cents         = 平台/商家优惠（仅记录，已扣在 gross 内不重复扣）
实收        received_amount_cents  = gross − refund − platform_fee − service_fee
                                     （以「交易账单」为准；缺时用「订单明细.预估到账」）
成本        cogs_cents             = unit_cost_cents × quantity（取下单时 balance.avg_cost_cents 作为对外成本快照）
净销售额    net_sales_cents        = gross − refund
毛利        profit_cents           = received_amount_cents − cogs_cents
毛利率      profit_rate            = profit_cents / received_amount_cents（received=0 时为 null）
```

**改造点**：

1. `migrations/0011_money_columns_align.sql`：
   - 给 `sales_orders` 补齐：`platform_fee_cents` / `service_fee_cents` / `discount_cents` / `refund_amount_cents`（如未存在）
   - 给 `sales_items` 补：`unit_cost_cents`、`line_cogs_cents` 已有则跳过
   - 索引：`sales_orders(record_date, type, machine_id)`

2. 新建 `functions/api/_shared/money.js` 导出上述常量与计算函数。
   - `computeOrderTotals(rows)` — 给一组明细行算汇总
   - `computeProfit({ received, cogs })` — 单一公式
   - `computeReceivedFromBill(bill)` — 从「交易账单」单行算实收

3. 新建 `functions/api/integrations/zn/import-settlement.js`：
   - 入参：交易账单 Excel 解析后的行数组
   - 字段映射见 skill `zn-excel-import`
   - 按 `订单号` UPSERT 进 `sales_orders`：只更新 `platform_fee_cents` / `service_fee_cents` / `received_amount_cents`，不动 quantity
   - 幂等键：`(integration='zn', external_id=订单号, settlement=true)` 写入 `external_settlement_imports`（新表，见迁移 0011）
   - 不存在的订单号 → warning，不创建

4. 前端 `frontend/app/components/settings/ZnImportCard.vue`：
   - 复用现有上传组件，新增「结算（交易账单）」tab
   - 解析逻辑统一抽到 `frontend/app/composables/useZnExcel.ts`

5. 报表：`functions/api/reports/dashboard.js` 改用 `_shared/money.js` 的公式，禁止再自拼 SQL 减式。

#### 4.4.2 新建迁移

**`migrations/0012_split_combined_machine.sql`**（不可逆，故必须先备份）：

```sql
BEGIN TRANSACTION;

-- 1. 把 stock_movements 上 '1/2号机' / '1/2号机总库存' 标为待重建
UPDATE stock_movements SET machine_id = '__pending_split__'
WHERE machine_id IN ('1/2号机', '1/2号机总库存');

-- 2. inventory_balances 上同名也清空（要由 importer 重新生成）
DELETE FROM inventory_balances WHERE machine_id IN ('1/2号机', '1/2号机总库存');

-- 3. products.machine_id 的 '1/2号机' → 沿用 D1 决策：保留为商品定义机型，
--    但库存按 sales_items / purchase_items 携带的 order.machine_id 去重建
--    （不修改 products.machine_id，避免误伤其它逻辑）

COMMIT;
```

**`migrations/0013_stock_transfers.sql`**（D3=A 才需要）：

```sql
CREATE TABLE IF NOT EXISTS stock_transfers (
  id TEXT PRIMARY KEY,
  product_id TEXT NOT NULL,
  from_machine_id TEXT NOT NULL,
  to_machine_id TEXT NOT NULL,
  quantity INTEGER NOT NULL CHECK (quantity > 0),
  unit_cost_cents INTEGER NOT NULL,
  reason TEXT,
  created_at TEXT NOT NULL,
  voided_at TEXT
);
CREATE INDEX IF NOT EXISTS idx_stock_transfers_product ON stock_transfers(product_id);
CREATE INDEX IF NOT EXISTS idx_stock_transfers_machines ON stock_transfers(from_machine_id, to_machine_id);
```

#### 4.4.3 新建 `_shared/inventory-balance.js`（合并 R1 三处副本）

```
exports:
  applyBalanceDelta(env, ctx)
    - ctx: { product_id, machine_id, qty_delta, unit_cost_cents, timestamp }
    - 读 balance、按权平均算新值、写回
    - 只用一种负库存恢复策略（保留：当 prevQty < 0 且 delta > 0 时，重置 inventory_value 为 max(0, qty * unit_cost)）
  rebuildBalanceFromMovements(env, product_id, machine_id)
    - 从 stock_movements 全量重算 balance，写回
    - Phase 2 重建主力 + 任何时候人工想重算都跑这一份
  applyTransfer(env, ctx)
    - 调用 applyBalanceDelta 两次（出 + 入）
```

`inventory-service.js` / `zn/importer.js` / `shengma/importer.js` 三处调用方全部改成 `applyBalanceDelta`，删除自己的副本。

#### 4.4.4 重建脚本 `scripts/rebuild-balances.mjs`

```
为每个 (product_id, machine_id)：
  1) DELETE FROM inventory_balances WHERE product_id = ? AND machine_id = ?
  2) 按 stock_movements.created_at ASC 顺序，对每条 movement 调 applyBalanceDelta
  3) 最后一次的 balance 即为最终值
  4) 全程在事务里
```

可任意时刻重跑，结果幂等。
